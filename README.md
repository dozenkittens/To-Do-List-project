# Техническое задание на разработку информационной системы
## «Планировщик задач (To-Do List)»

Версия 1.0

---

### 1. Общие сведения

**1.1. Наименование системы**
Полное наименование: Информационная система «Планировщик задач».
Краткое наименование: SmartTodo.

**1.2. Назначение системы**
Система предназначена для автоматизации процесса планирования личных задач и контроля их выполнения. Программный продукт позволяет пользователю создавать перечень задач, отслеживать их статус и получать наглядную статистику о прогрессе.

**1.3. Цели создания системы**
- Разработка удобного инструмента для повседневного планирования задач и повышения личной продуктивности пользователя
- Обеспечение быстрого доступа к списку задач с возможностью управления ими (добавление, выполнение, удаление)
- Визуализация прогресса выполнения задач для мотивации пользователя
- Получение практических навыков проектирования и разработки программных модулей студентом

**1.4. Целевая аудитория**
- Студенты и школьники для отслеживания учебных заданий
- Работающие люди для управления рабочими и личными задачами
- Все пользователи, желающие повысить личную продуктивность

**1.5. Основные сценарии использования**

| № | Сценарий | Действие пользователя | Результат |
|:--:|----------|----------------------|-----------|
| 1 | Регистрация | Пользователь заполняет форму регистрации | Создается новый аккаунт |
| 2 | Вход в систему | Пользователь вводит логин и пароль | Открывается главная страница с задачами |
| 3 | Создание задачи | Пользователь вводит название задачи и нажимает "Добавить" | Задача появляется в списке |
| 4 | Отметка выполнения | Пользователь нажимает на чекбокс задачи | Задача отмечается выполненной, обновляется статистика |
| 5 | Удаление задачи | Пользователь нажимает на иконку корзины | Задача удаляется из списка |
| 6 | Просмотр статистики | Пользователь открывает главную страницу | Отображается дашборд с показателями |

### 2. Функциональные требования

#### 2.1. Основные функции системы

Система должна обеспечивать выполнение следующих функций:

**2.1.1. Управление пользователями**
- Регистрация нового пользователя
- Аутентификация пользователя (вход в систему)
- Выход из системы

**2.1.2. Управление задачами**
- Создание новой задачи (пользователь должен указать название задачи)
- Просмотр полного списка всех созданных задач
- Редактирование задачи (изменение названия)
- Удаление задачи

**2.1.3. Учет выполнения задач**
- Отметка задачи как выполненной (нажатие на чекбокс)
- Отмена отметки о выполнении (снятие чекбокса)
- Визуальное различие выполненных и активных задач (зачеркивание текста)

**2.1.4. Статистика и аналитика**
- Подсчет и отображение общего количества задач
- Подсчет и отображение количества выполненных задач
- Подсчет и отображение количества активных (невыполненных) задач

#### 2.2. CRUD-операции для основных сущностей

| Сущность | Create (Создание) | Read (Чтение) | Update (Обновление) | Delete (Удаление) |
|----------|:-----------------:|:-------------:|:-------------------:|:-----------------:|
| Пользователь | Регистрация | Получение профиля | Обновление профиля | Удаление аккаунта |
| Задача | Создание задачи | Получение списка задач | Отметка выполнения / Редактирование | Удаление задачи |

### 3. Архитектура системы

#### 3.1. Клиент-серверная архитектура

Система реализована по клиент-серверной архитектуре с разделением на frontend и backend компоненты.

┌─────────────────┐ ┌─────────────────┐ ┌─────────────────┐
│ │ │ │ │ │
│ Frontend │ ◄─────► │ Backend │ ◄─────► │ Database │
│ (React/JS) │ HTTP │ (API Server) │ SQL │ (PostgreSQL) │
│ │ API │ │ │ │
└─────────────────┘ └─────────────────┘ └─────────────────┘

#### 3.2. Взаимодействие frontend и backend

- Frontend отправляет HTTP-запросы к backend API
- Backend обрабатывает запросы, взаимодействует с базой данных и возвращает ответ в формате JSON
- Взаимодействие происходит асинхронно через HTTP/HTTPS протокол

#### 3.3. Тип API

В системе используется **REST API** (Representational State Transfer), который соответствует следующим принципам:
- Клиент-серверное разделение
- Отсутствие состояния (stateless)
- Единообразие интерфейса
- Использование HTTP-методов для операций с ресурсами

### 4. Описание API

#### 4.1. Базовый URL
https://api.smarttodo.com/api/v1


#### 4.2. HTTP-методы и их назначение

| Метод | Назначение | Пример использования |
|:-----:|------------|----------------------|
| GET | Получение данных | Получение списка задач |
| POST | Создание нового ресурса | Создание новой задачи |
| PUT / PATCH | Обновление существующего ресурса | Отметка задачи как выполненной |
| DELETE | Удаление ресурса | Удаление задачи |

#### 4.3. Эндпоинты API

**Аутентификация и пользователи**

| Эндпоинт | Метод | Описание | Тело запроса | Ответ |
|----------|:-----:|----------|--------------|-------|
| `/auth/register` | POST | Регистрация пользователя | `{ "username": "Alex", "email": "alex@mail.com", "password": "123456" }` | `{ "id": 1, "username": "Alex", "token": "jwt_token..." }` |
| `/auth/login` | POST | Вход в систему | `{ "email": "alex@mail.com", "password": "123456" }` | `{ "user": {...}, "token": "jwt_token..." }` |
| `/auth/logout` | POST | Выход из системы | `{ "token": "jwt_token..." }` | `{ "message": "Logged out successfully" }` |

**Задачи**

| Эндпоинт | Метод | Описание | Тело запроса | Ответ |
|----------|:-----:|----------|--------------|-------|
| `/tasks` | GET | Получение всех задач пользователя | — | `{ "tasks": [{"id": 1, "title": "...", "is_completed": false}] }` |
| `/tasks` | POST | Создание новой задачи | `{ "title": "Новая задача" }` | `{ "id": 1, "title": "Новая задача", "is_completed": false }` |
| `/tasks/{id}` | PUT | Обновление задачи | `{ "title": "Обновленное название", "is_completed": true }` | `{ "id": 1, "title": "...", "is_completed": true }` |
| `/tasks/{id}` | DELETE | Удаление задачи | — | `{ "message": "Task deleted successfully" }` |

#### 4.5. Коды ответов HTTP

| Код | Значение | Описание |
|:---:|----------|----------|
| 200 | OK | Запрос успешно выполнен |
| 201 | Created | Ресурс успешно создан |
| 400 | Bad Request | Неверный запрос (ошибка валидации) |
| 401 | Unauthorized | Не авторизован (требуется токен) |
| 403 | Forbidden | Доступ запрещен (недостаточно прав) |
| 404 | Not Found | Ресурс не найден |
| 500 | Internal Server Error | Ошибка на сервере |

#### 4.6. Аутентификация пользователей

В системе используется JWT (JSON Web Token) аутентификация:

1. Пользователь отправляет логин и пароль на эндпоинт `/auth/login`
2. Сервер проверяет учетные данные и возвращает JWT-токен
3. Клиент сохраняет токен (в localStorage или httpOnly cookies)
4. При каждом запросе к API клиент добавляет токен в заголовок:
Authorization: Bearer <jwt_token>


5. Сервер проверяет валидность токена перед обработкой запроса

#### 4.7. Авторизация пользователей

- Каждый пользователь имеет доступ только к своим задачам
- При создании задачи автоматически устанавливается user_id из токена
- При запросе списка задач возвращаются только задачи текущего пользователя
- При попытке доступа к чужой задаче возвращается ошибка 403 Forbidden
Пользователь A → может видеть/редактировать/удалять только свои задачи
Пользователь B → может видеть/редактировать/удалять только свои задачи

**2. Дашборд (Блок статистики)**

Три информационные карточки, расположенные горизонтально в ряд:

- Карточка «Всего задач» — отображает общее количество задач (иконка 📋)
- Карточка «Выполнено» — отображает количество выполненных задач (иконка ✅)
- Карточка «Активно» — отображает количество невыполненных задач (иконка ⏳)

Каждая карточка содержит иконку, название показателя и числовое значение.

**3. Блок создания задачи**

- Текстовое поле для ввода названия новой задачи (placeholder: «Новая задача...»)
- Кнопка «+ Добавить», расположенная справа от поля ввода. Нажатие кнопки добавляет задачу в список

**4. Список задач**

Область для отображения всех созданных задач в виде вертикального списка.

Каждый элемент списка (отдельная задача) содержит:

- Чекбокс (квадратный элемент) слева для отметки выполнения
- Текст задачи в центре элемента
- Иконка «Корзина» (🗑️) справа для удаления задачи

**5. Визуальные состояния задач**

| Состояние | Чекбокс | Текст | Действие |
|-----------|:-------:|-------|----------|
| Активная задача | ⬜ | Обычный (не зачеркнутый) | Можно отметить выполнение |
| Выполненная задача | ✅ | Зачеркнутый, серым цветом | Можно снять отметку |

### 6. Состав и структура работ

Разработка системы осуществляется в несколько этапов:

1. Проектирование структуры системы и написание технического задания
2. Разработка UX/UI дизайна (создание структуры страницы из примитивов)
3. Верстка пользовательского интерфейса (HTML, CSS)
4. Программирование логики приложения (JavaScript)
5. Разработка backend API (Node.js/Express)
6. Подключение базы данных (PostgreSQL)
7. Интеграция frontend с backend API
8. Тестирование и отладка
9. Подготовка отчетной документации

### 7. Порядок контроля и приемки

Приемка системы осуществляется преподавателем по результатам выполнения всех этапов работ. Система считается принятой, если:

- Весь заявленный функционал (раздел 2) работает без ошибок
- Интерфейс соответствует утвержденному UX-дизайну (раздел 5.2)
- API эндпоинты (раздел 4) работают корректно и возвращают ожидаемые ответы
- Реализована аутентификация и авторизация пользователей
- Код программы является читаемым и структурированным
